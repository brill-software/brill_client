The Brill CMS supports the development of real-time e-commerce and e-learning websites

Brill CMS

An application is composed of pages, with each page composed of components. The components make use of various resources which are also part of the application. The pages and often resources are encoded in JSON. The Brill CMS supports editing pages and resources that comprise an application.

Pages can be edited using a JSON or WYSIWYG editor. The JSON editor is more suited to developers that want to know exactly what the page structure is and the attributes that are passed to components. The WYSISYG editor is more suited to business users and content developers that want to concentrate on the page appearance, rather than the underlying technical details. Users can easily switch between the different edit modes.

To use the CMS, the user needs to login. The user is presented with a list of applications that they own or have permission to access. The user can also create a new application or copy an existing application.

Once an application is selected for editing, there’s a tree on the left hand side with the pages, files and directory structure. The user can select a page or file to edit. With pages, there are number of different edit modes. Files are edited using a text editor with syntax colour highlighting. The CMS supports both a dark theme and light theme.

Themes

A Theme allows a consistent set of colours, fonts and styling to be applied across an application. Normally a Theme will be developed that matches the organisations style guide for colours, fonts and layout. This can then be applied across all the organisations applications.

Most of the Brill Components are based on Material UI components. There’s a default Material UI Theme which is documented at:

https://material-ui.com/customization/default-theme/

The MUI Theme has been extended to support some additional fields. The override section can be used to override Brill Components, as well as MUI Components. There are also two additional sections that allow the styles of the <html> and <body> tags to be specified.

This is an example of a Theme file.

It sets the primary colour for use by MUI Components. The <h1> styling is specified for both large screens and small screens. The styling for the MenuButton components is specified and also the styling for the page <body> tag.

Best practice is to use themes for styling. You can however set the style of an individual component using the style attribute. This is useful where a component requires a slightly different style from that specified in the theme.

Material UI Components

Most of the Brill UI components are based on Material-UI components. Material-UI is an open source framework that implements Google’s Material Design. It's the world's most popular React UI framework.

Custom components can be developed using other frameworks such as Semantic-UI if there’s a requirement to do so.

Component Palette

The Component Palette contains components that can be dragged and dropped onto the page. The Component Palette can be customised to add or remove particular components. This is useful to provide Page Editors with the components that just match the organisations style guide. Custom developed components can be added to the palette and the defaults for various attributes set.

Media Library

The library is shared across applications and holds images, themes, templates and other resources that could be useful. Images and themes can be copied from the library into an application. Library items can be hidden so that just the items that match the organisations style guide are displayed.

Page Templates

A set of template pages is provided to support quick page creation. Additional templates can be added. Templates can also be hidden. Pages can include common sections for items like headers and footers.

Permissions

A users has to be granted access to edit an application by the application owner. The application owner is the user that originally created the application. Ownership can be transferred to another user and also shared.

An application owner has full read-write access to the application and can publish a new version of the application. An application owner can grant access to other users. The levels that can be granted are:

Application Owner

Publish rights

Full read/write access

Editing of text and image components only

Read only access but with ability to add comments

Git repository

All application files are held in a git repository. Best practice is to use a MASTER branch, which is the Production branch and a DEVELOPMENT branch. Changes are made and tested on the DEVELOPMENT branch and when ready, merged into the MASTER branch. Changes are not normally made on the MASTER branch. The history of all files can be viewed. When changes are saved, the user is prompted to enter a description of the change. The description is saved in git.
